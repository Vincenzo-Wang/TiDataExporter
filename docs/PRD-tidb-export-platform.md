**项目名称：** Claw Export Platform
**PRD 版本：** 1.0.0
**创建日期：** 2026-03-23
**作者：** Product Team

## Problem Statement

### 业务背景

支付系统已全面使用 TiDB 数据库作为核心数据存储。随着业务的发展，运营、财务、数据分析等团队需要频繁导出业务数据，包括订单记录、用户数据、交易流水等。目前的数据导出方式存在以下问题：

1. **分散的导出方式：** 每个业务系统各自实现导出功能，缺乏统一标准
2. **重复开发：** 每个项目都需要独立开发导出功能，造成资源浪费
3. **管理困难：** 缺乏统一的任务管理和监控，难以跟踪导出任务状态
4. **安全隐患：** 数据导出缺乏统一的安全管控和审计机制
5. **扩展性差：** 无法快速适配新增的业务系统和数据源

### 核心问题

从不同用户的角度来看，当前面临的问题包括：

**运营/财务人员：**
- 需要定期导出报表数据，但需要依赖开发人员手动执行 SQL 并发送文件
- 无法自助导出数据，流程繁琐且耗时
- 导出的文件分散存储，难以追溯历史记录

**业务系统开发者：**
- 每个项目都需要开发导出功能，重复劳动
- 缺乏统一的导出工具和配置规范
- 需要自己处理文件存储和分发逻辑

**系统管理员：**
- 缺乏统一的导出任务监控和告警机制
- 无法有效管控数据导出的权限和配额
- 存储成本难以控制，过期文件无法自动清理

**数据安全：**
- 多租户数据缺乏隔离机制
- 导出操作缺乏审计追踪
- 敏感数据缺乏加密和脱敏保护

### 目标

构建一个通用的 TiDB 数据导出平台，为所有使用 TiDB 的业务系统提供统一的数据导出服务，实现：

1. **统一管理：** 集中管理所有导出任务、配置和权限
2. **自助服务：** 业务系统通过 API 自助创建和查询导出任务
3. **安全可靠：** 多租户数据隔离，完善的认证鉴权和审计机制
4. **易于扩展：** 快速适配新增的业务系统，无需重复开发
5. **成本可控：** 自动清理过期文件，有效控制存储成本

## Solution

### 整体解决方案

构建一个基于 Go + React 的 TiDB 数据导出平台，采用异步任务队列架构，支持多租户独立配置和管理。平台包含以下核心能力：

### 1. 多租户管理

**租户隔离：**
- 每个业务系统（租户）独立配置 TiDB 连接和 S3 存储
- 租户数据完全隔离，通过 tenant_id 进行严格的数据访问控制
- 支持为每个租户配置独立的配额限制（并发任务数、每日任务数、导出数据量）

**租户配置：**
- TiDB 连接配置：支持多套配置，可设置默认配置
- S3 存储配置：支持多套配置，可设置默认配置，支持自定义路径前缀
- Dumpling 模板：预设常用导出参数模板，简化任务创建

### 2. 开放 API

**任务生命周期管理：**
- 创建导出任务：接收 SQL 语句，配置导出参数，返回任务 ID
- 查询任务状态：轮询查询任务状态，获取执行进度和结果
- 取消任务：取消正在执行或等待中的任务
- 批量查询：批量查询多个任务的状态，提高查询效率

**文件访问：**
- 直接下载：返回 S3 预签名 URL，业务系统可直接下载
- 代理下载：通过平台代理下载，避免 URL 泄露

**认证方式：**
- API Key + API Secret：用于业务系统调用
- 支持 HMAC 签名增强安全性
- IP 白名单限制（可选）

### 3. 管理后台

**任务管理：**
- 任务列表：分页查看所有任务，支持按租户、状态、时间筛选
- 任务详情：查看任务的完整信息、执行日志、错误信息
- 任务操作：取消任务、重试失败任务、手动清理文件

**日志与监控：**
- 执行日志：查看任务的详细执行日志（Dumpling 输出、系统日志）
- 统计报表：按租户、时间、状态统计任务成功率和数据量
- 系统监控：查看 Worker 状态、队列积压、资源使用情况

**配置管理：**
- 租户管理：创建、编辑、删除租户，重置 API Key
- TiDB 配置管理：创建、编辑、删除 TiDB 连接配置
- S3 配置管理：创建、编辑、删除 S3 存储配置
- Dumpling 模板管理：创建、编辑、删除导出参数模板

**管理员与权限：**
- 管理员登录：用户名 + 密码 + JWT Token 认证
- 权限控制：基于角色的访问控制（RBAC）
  - admin：所有权限
  - operator：查看任务、查看日志、创建配置，不能删除租户

### 4. 异步任务处理

**任务执行流程：**
1. API Gateway 接收任务创建请求，验证参数和配额
2. 创建任务记录（status=pending），推送任务到 Redis Stream
3. Worker Pool 从队列获取任务，更新状态为 running
4. Worker 调用 Dumpling 执行导出，实时捕获日志
5. Worker 上传文件到 S3，记录文件信息和任务状态
6. 更新任务状态为 success/failed，ACK 消息

**任务管理机制：**
- 取消机制：支持取消 pending/running 状态的任务，终止 Dumpling 进程
- 重试机制：失败任务自动重试（最多 3 次），指数退避策略
- 超时机制：超时任务自动终止（默认 2 小时，可配置）
- 幂等性保证：Worker 崩溃恢复后，已完成的任务不会重复执行

**失败处理：**
- Dumpling 执行失败：清理临时文件，记录错误日志
- S3 上传失败：保留本地文件，允许手动重试
- 配置错误：立即失败，不进行重试
- SQL 语法错误：立即失败，不进行重试

### 5. 文件生命周期管理

**文件上传：**
- Dumpling 输出到本地目录或直接输出到 S3
- 如果输出到本地，使用 AWS SDK 上传到 S3
- 支持 S3 分片上传，优化大文件传输性能
- 文件命名规则：`{tenant_id}/{task_id}/{timestamp}_{filename}.{ext}`

**文件访问：**
- 生成 S3 预签名 URL，有效期与文件保留时间一致
- 支持 HTTP/HTTPS 访问
- URL 包含签名，防止未授权访问

**文件清理：**
- 定时任务每小时执行一次，清理过期文件
- 删除 S3 上的过期文件
- 更新任务状态为 expired
- 清理本地临时目录
- 支持手动触发清理

**文件保留策略：**
- 业务系统可指定文件保留时间（默认 168 小时，即 7 天）
- 租户配额可限制最长保留时间
- 过期文件自动删除，无法恢复

### 6. 安全机制

**认证与授权：**
- API Key 认证：API Key + API Secret，Secret 使用 AES-256 加密存储
- JWT Token 认证：管理后台使用 JWT Token，有效期 2 小时
- RBAC 权限控制：基于角色的访问控制，区分 admin 和 operator
- CSRF 防护：管理后台所有写操作需要 CSRF Token

**数据安全：**
- 敏感信息加密：API Secret、数据库密码、S3 Secret Key 使用 AES-256 加密存储
- 密码哈希：管理员密码使用 bcrypt 哈希存储
- SQL 注入防护：使用 SQL 解析器验证 SQL 语句，只允许 SELECT 语句
- 敏感字段脱敏：API 响应中不返回完整的 SQL，只返回摘要

**审计日志：**
- 记录所有敏感操作：登录/登出、配置变更、任务创建/取消、租户管理
- 审计日志包含：操作人、操作类型、操作对象、请求 IP、操作时间、操作结果
- 审计日志持久化存储，支持查询和导出

**限流与配额：**
- API 限流：基于 Redis 令牌桶算法，每个租户独立限流（默认 100 req/min）
- 任务并发限制：每个租户最大并发任务数（默认 5 个）
- 资源配额：每日最大任务数、每日最大导出数据量
- IP 白名单：可限制租户的访问 IP 地址段

### 7. 监控与运维

**监控指标：**
- 业务指标：任务成功率、执行时长、排队时间、导出数据量、API QPS
- 系统指标：Worker 在线数量、CPU/内存使用率、任务队列积压量、数据库连接数
- 存储指标：S3 存储容量、文件数量、过期文件清理情况

**告警规则：**
- P0 紧急：服务宕机、数据库连接失败、任务失败率 > 50%（持续 5 分钟）
- P1 重要：Worker 数量 < 最小阈值、任务队列积压 > 1000 个、API 错误率 > 5%
- P2 一般：任务执行时长 > 1 小时、API 响应时间 > 5s（P95）

**日志规范：**
- 结构化日志：JSON 格式，包含 timestamp、level、service、tenant_id、task_id、message
- 日志级别：DEBUG、INFO、WARN、ERROR
- 日志脱敏：不输出敏感信息（密码、Secret Key、完整 SQL）

**健康检查：**
- `/health`：基础健康检查（数据库、Redis、S3 连接状态）
- `/health/ready`：就绪检查（Worker 状态、任务队列积压量）

### 8. 技术架构

**技术栈：**
- 后端：Go + Gin + GORM + go-redis + AWS SDK for Go v2
- 前端：React 18 + TypeScript + Ant Design + Zustand + React Query
- 消息队列：Redis Stream
- 数据库：TiDB（元数据库）
- 存储：S3 兼容对象存储

**部署模式：**
- 单进程部署：API Gateway 和 Worker Pool 在同一进程（开发/小规模生产）
- 多进程部署：API Gateway 和 Worker Pool 分别部署（大规模生产）
- 支持水平扩展：API Gateway 无状态，Worker Pool 支持多个实例
