## 实现决策

### 1. 架构决策

**1.1 异步任务队列架构**
- 使用 Redis Stream 作为消息队列，Worker Pool 异步处理导出任务
- 理由：避免同步处理导致超时，Redis Stream 支持消费者组和故障恢复

**1.2 多租户架构**
- 所有数据表包含 tenant_id，查询自动注入租户过滤条件
- 理由：支持多个业务系统独立使用，数据完全隔离

**1.3 Go + React 技术栈**
- 后端：Go + Gin + GORM + go-redis
- 前端：React 18 + TypeScript + Ant Design + Zustand
- 理由：团队熟悉，性能优秀，生态完善

**1.4 支持单进程和多进程部署**
- 单进程：API Gateway 和 Worker Pool 在同一进程（开发/小规模）
- 多进程：API Gateway 和 Worker Pool 分别部署（大规模生产）

### 2. 数据模型决策

**2.1 API Secret 使用 AES-256 加密**
- tenants.api_secret_encrypted 使用 AES-256 加密存储
- 理由：API Secret 需要还原才能验证，加密而非哈希

**2.2 管理员密码使用 bcrypt 哈希**
- admins.password_hash 使用 bcrypt 哈希存储
- 理由：密码不需要还原，bcrypt 是业界标准

**2.3 所有表增加 deleted_at 软删除**
- 支持软删除，便于审计和恢复
- 理由：保留历史记录，避免级联删除导致的数据丢失

**2.4 export_tasks 增加优先级和重试字段**
- 增加 priority、retry_count、max_retries 字段
- 理由：支持任务优先级和自动重试机制

**2.5 增加 tenant_quotas 表**
- 记录每个租户的配额限制
- 理由：防止单租户占用过多资源

**2.6 增加 audit_logs 表**
- 记录所有敏感操作
- 理由：安全设计提到审计日志，需要有对应的表

### 3. API 设计决策

**3.1 开放 API 使用 API Key 认证**
- HTTP Header X-API-Key 和 X-API-Secret
- 理由：简单易用，适合业务系统集成

**3.2 管理 API 使用 JWT Token 认证**
- Bearer Token，有效期 2 小时
- 理由：无状态，适合 Web 应用

**3.3 补充完整的配置管理 API**
- 租户管理、TiDB 配置、S3 配置、Dumpling 模板
- 理由：管理后台需要这些接口才能工作

**3.4 补充任务取消、批量查询、文件下载代理 API**
- DELETE /api/v1/export/tasks/{task_id}
- POST /api/v1/export/tasks/batch
- GET /api/v1/export/files/{task_id}
- 理由：业务系统需要这些功能

**3.5 补充健康检查和就绪检查 API**
- /health 和 /health/ready
- 理由：用于负载均衡和滚动更新

### 4. 业务流程决策

**4.1 任务取消流程**
- pending：直接更新状态为 canceled
- running：发送取消信号给 Worker，Worker 终止 Dumpling 进程
- completed：返回错误

**4.2 任务重试流程**
- 检查 retry_count < max_retries
- 判断是否可重试（Dumpling 失败、S3 上传失败可重试）
- 指数退避重试（2^retry_count 秒）

**4.3 任务失败清理流程**
- Dumpling 失败：清理本地输出目录
- S3 上传失败：保留本地文件，允许手动重试

**4.4 Worker 崩溃恢复流程**
- Redis Stream 消息超时（30 分钟）后重新进入 Pending
- 幂等性保证：检查 S3 是否已有文件，有则直接更新状态为 success

**4.5 任务超时处理流程**
- 定时任务每分钟执行一次
- 查询超时任务（running 且 started_at < 2 小时前）
- 发送终止信号，更新状态为 failed

### 5. 安全设计决策

**5.1 SQL 注入防护**
- 使用 SQL 解析器（vitess）验证 SQL 语法
- 只允许 SELECT 语句
- 拒绝包含修改操作关键字的 SQL

**5.2 敏感字段脱敏**
- API 响应中不返回完整的 sql_text，只返回摘要（前 100 字符）

**5.3 S3 访问控制**
- 生成预签名 URL，有效期与文件保留时间一致
- 每个 IP 每分钟最多下载 10 次

**5.4 密钥轮换**
- 定期轮换加密密钥（建议每 6 个月）
- API Secret 和管理员密码可手动重置

## 测试决策

### 1. 测试原则

**1.1 只测试外部行为，不测试实现细节**
- 测试关注系统对外提供的功能和接口
- 避免测试内部实现细节，导致测试脆弱

**1.2 测试应该独立且可重复**
- 每个测试独立运行，不依赖其他测试的状态
- 可以重复运行得到相同结果

**1.3 测试应该覆盖正常路径和异常路径**
- 覆盖正常的业务流程
- 覆盖异常情况和边界条件

**1.4 测试应该快速且稳定**
- 使用内存数据库进行单元测试
- Mock 外部依赖以加快测试速度

- Model 层：数据库操作、数据验证
- Worker 层：任务处理、错误处理、幂等性
- 中间件：认证、鉴权、限流
- 工具函数：加密、解密、SQL 解析、日志格式化

**2.2 测试覆盖率要求**
- 总体覆盖率：≥ 80%
- 核心模块覆盖率：≥ 90%（认证、鉴权、任务处理、SQL 解析）

### 3. 集成测试

## 超出范围

### 1. 功能范围

**不包含以下功能：**
1. **定时任务配置**：在管理后台配置定时导出任务，支持 Cron 表达式
2. **数据转换**：导出后进行数据转换、格式转换、数据脱敏
3. **通知机制**：任务完成邮件通知、Webhook 回调、钉钉/企业微信集成
4. **OAuth2 认证**：支持 OAuth2 + JWT，集成企业 SSO，多因素认证
5. **高级调度功能**：任务优先级（虽已预留字段，但实现优先级队列是可选的）、任务依赖、延迟队列
6. **分布式任务调度**：使用分布式任务调度框架（如 Temporal），支持任务依赖和编排

### 2. 技术范围

**不包含以下技术：**
1. **消息队列升级**：从 Redis Stream 迁移到 RabbitMQ/Kafka
2. **微服务化**：拆分为 API Gateway、Worker、配置服务等独立服务，使用 Service Mesh
3. **数据库迁移**：从 TiDB 迁移到其他数据库
4. **高级监控**：使用分布式追踪（Jaeger、Zipkin）、APM 工具（New Relic、Datadog）

### 3. 部署范围

**不包含以下部署方案：**
1. **多云部署**：同时部署在多个云服务提供商（AWS、阿里云、腾讯云）
2. **容器编排**：使用 Kubernetes 编排容器，自动扩缩容
3. **灾备方案**：跨数据中心灾备、数据异地备份

### 4. 数据范围

**不包含以下数据处理：**
1. **实时数据处理**：实时流式导出、增量导出
2. **大数据处理**：超过 10GB 的大数据导出、分布式导出
3. **数据归档**：将导出的数据归档到冷存储、数据生命周期管理

## 其他说明

### 1. 非功能性需求

**1.1 可用性**
- 系统可用性：≥ 99.5%（月度）
- 故障恢复时间：< 30 分钟（P0 级别故障）
- 数据持久性：≥ 99.9%（审计日志）

**1.2 可扩展性**
- 水平扩展：支持 API Gateway 和 Worker Pool 的水平扩展
- 垂直扩展：支持增加单实例的资源（CPU、内存）
- 存储扩展：S3 自动扩展容量，无需人工干预

**1.3 性能**
- API 响应时间：P50 < 100ms, P95 < 500ms, P99 < 1000ms
- 任务执行时间：小任务 1-5 分钟，中等任务 5-15 分钟，大任务 15-60 分钟
- 并发能力：单实例 4-8 任务/小时，水平扩展后 50+ 任务/小时

**1.4 安全性**
- 认证：API Key 认证 + JWT Token 认证
- 加密：敏感信息使用 AES-256 加密存储，传输使用 HTTPS
- 审计：所有敏感操作记录审计日志
- 限流：基于 Redis 令牌桶算法，防止 API 滥用

**1.5 可维护性**
- 日志：结构化日志（JSON 格式），便于查询和分析
- 监控：完善的监控指标和告警规则
- 文档：API 文档、部署文档、运维手册
- 代码质量：代码审查、单元测试覆盖率 ≥ 80%

### 2. 约束和假设

**2.1 约束**
- 时间约束：需要在 10 周内完成 MVP 版本
- 资源约束：开发团队 2 人（后端）+ 1 人（前端）
- 技术约束：必须使用 Go + React 技术栈
- 环境约束：支持部署在 Docker 容器中

**2.2 假设**
- 数据量：当前主要处理中等数据量（100MB-5GB），未来可能扩展到大数据量
- 租户数量：当前主要服务 3-5 个租户，未来可能扩展到 10+ 个租户
- QPS：当前 API QPS < 100，未来可能增长到 1000+
- 可用性：当前要求 99.5%，未来可能提高到 99.9%

### 3. 风险和依赖

**3.1 技术风险**
| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| TiDB 性能瓶颈 | 高 | 中 | 使用连接池、优化查询、增加缓存 |
| Dumpling 执行失败 | 高 | 低 | 完善错误处理、支持重试、记录详细日志 |
| S3 上传失败 | 高 | 低 | 分片上传、断点续传、重试机制 |
| Redis 单点故障 | 中 | 低 | 使用 Redis Cluster 或哨兵模式 |

**3.2 业务风险**
| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| 任务积压 | 中 | 低 | 动态扩容 Worker、限流保护 |
| 存储成本过高 | 中 | 中 | 定期清理过期文件、压缩存储 |
| 数据泄露 | 高 | 低 | 加密存储、访问控制、审计日志 |

**3.3 依赖关系**
- 外部依赖：TiDB（元数据库和业务数据库）、Redis（消息队列和缓存）、S3 兼容对象存储（文件存储）、Dumpling CLI（数据导出工具）
- 内部依赖：业务系统需要集成导出平台的 API，运维团队需要部署和维护导出平台
